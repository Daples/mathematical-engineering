Programming Lab 2: Git Commits in a Repository

TA: Juan Sebastián Cárdenas-Rodríguez

Introduction

Git is a distributed version-control system for tracking changes in source code during software development. This tool was created by Linus Torvalds and it can be used jointly with other hosting services such as GitHub. Furthermore, command-line operations as commit, push, pull and so fourth are one of the main features of Git that allows for an easier and flexible source code management and control.

In this programming lab you are expected to extract some attributes for each commit done in the Atom project, and save it in a CSV file to better process this information.

To extract the commits in the desired format, clone the Github repository of the Atom project and run the following command in a terminal inside the cloned repository:

git rev-list --all --pretty=raw --dense --no-abbrev > atomCommits.txt

Assignment (80%)

Using all commits available in the Atom GitHub repository write a program that does the following:

  • (35%) Read the file atomCommits.txt, parse it, and create a CSV file atomCommits.csv with the format:

    <Author>,<Committer>,<Author Email>,<Commit ID>,<Parent>,<GPG Signed?>,<Commit Message>

    If there is no information available for a given attribute the corresponding field should be left blank. Furthermore, if the email is a local address instead of an email account it should be also left in blank.

    You might find the libraries cassava and regex-tdfa useful.

  • (35%) Oral explanation of your solution.

  • (10%)
    • Your lab will be tested using the -Wall, -Wmissing-signatures and -Wmissing-local-signatures.
    • Document (in English) your source code. The documentation should explain your solution for the assignment.
    • Improve your implementation with HLint ≥ 3.2.1

  • The programming lab may either be solved on your own, or jointly with one other student taking the course.

  • You are to use GHC ≥ 8.10.1. See installing a proper Haskell distribution.

  • Does not have long lines (at most 80 columns).
  • Has an uniformly indentation (we recommend two characters).
  • Has a consistent layout.
  • Has type signatures for everything (included where-clauses and let-definitions).
  • Has good comments.
  • Has no junk (unused code, commented code, unnecessary code).
  • Has no overly complicated function definitions.
  • Does not contain any repetitive code.
  • Has no tabs.
  • Has no unnecessary spaces at the end of a line, or empty lines at the end of a file. (See: Fixing trailing whitespace issues with Emacs ).
